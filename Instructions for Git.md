<a name="history"></a>
## <p style="font-size: 30px;">Историческая справка</p>

---
###### <p align="right">[Cодержание](#content)</p>
<br>
<p style = text-indent:25px>Как и многие вещи в жизни, Git начинался с капелькой творческого хаоса и бурных споров.</p>

<p style = text-indent:25px>Ядро Linux — это достаточно большой проект с открытым исходным кодом. Большую часть времени разработки ядра Linux (1991–2002 гг.) изменения передавались между разработчиками в виде патчей и архивов. В 2002 году проект ядра Linux начал использовать проприетарную децентрализованную СКВ BitKeeper.</p>

<p style = text-indent:25px>В 2005 году отношения между сообществом разработчиков ядра Linux и коммерческой компанией, которая разрабатывала BitKeeper, прекратились, и бесплатное использование утилиты стало невозможным. Это сподвигло сообщество разработчиков ядра Linux (а в частности Линуса Торвальдса — создателя Linux) разработать свою собственную утилиту, учитывая уроки, полученные при работе с BitKeeper. </p>

![Линус Торвальдс негодует из-за BitKeeper](Linus-Torvalds.jpg)

<p style = text-indent:25px>Некоторыми целями, которые преследовала новая система, были:</p>

* Скорость
* Простая архитектура
* Хорошая поддержка нелинейной разработки (тысячи параллельных веток)
* Полная децентрализация
* Возможность эффективного управления большими проектами, такими как ядро Linux (скорость работы и разумное использование дискового пространства)



<p style = text-indent:25px>С момента своего появления в 2005 году, Git развился в простую в использовании систему, сохранив при этом свои изначальные качества. Он удивительно быстр, эффективен в работе с большими проектами и имеет великолепную систему веток для нелинейной разработки.</p>
<br><br>

<a name="git-command"></a>

## <p style="font-size: 30px;">Команды Git для работы внутри ветки.</p>
---
###### <p align="right">[Cодержание](#content)</p>
<br>
<a name="init"></a>

### <p style="font-size: 20px;">Начало работы - Инициализация.</p>

```
git init
```

<p style = text-indent:25px>Эта команда создаёт в текущем каталоге новый скрытый подкаталог с именем .git, содержащий все необходимые файлы репозитория — структуру Git репозитория. На этом этапе проект ещё не находится под версионным контролем.</p>

<br><a name="status"></a>

### <p style="font-size: 20px;">Отслеживание состояния Git</p>

```
git status
```

<p style = text-indent:25px>Команда отображает состояние рабочего каталога и раздела проиндексированных файлов. С ее помощью можно проверить индексацию изменений и увидеть файлы, которые не отслеживаются Git.</p>

<br><a name="add"></a>

### <p style="font-size: 20px;">Отслеживание файлов и коммиты</p>

```
git add имя_файла
```

Команда git add добавляет изменение из рабочего каталога в раздел проиндексированных файлов. Она сообщает Git, что вы хотите включить изменения в конкретном файле в следующий коммит.

<br><a name="commit"></a>

### <p style="font-size: 20px;">Сохранение и комментирование файла</p>

```
git commit -m "Текст комментария"
```

Регистрирует сделанные изменения в обозначенном файле и добавляет комментарий (описание изменений).

<br><a name="gitignore"></a>

### <p style="font-size: 20px;">Игнорирование файлов</p>

<p style = text-indent:25px>Игнорируемые файлы — это, как правило, артефакты сборки и файлы, генерируемые машиной из исходных файлов в вашем репозитории, либо файлы, которые по какой-либо иной причине не должны попадать в коммиты.</p>

<p style = text-indent:25px>Игнорируемые файлы отслеживаются в специальном файле .gitignore, который регистрируется в корневом каталоге репозитория. В Git нет специальной команды для указания игнорируемых файлов: вместо этого необходимо вручную отредактировать файл .gitignore, чтобы указать в нем новые файлы, которые должны быть проигнорированы. Файлы .gitignore содержат шаблоны, которые сопоставляются с именами файлов в репозитории для определения необходимости игнорировать эти файлы.</p>

Подробнее о формировании файла .gitignore можно почитать в статье ".gitignore" по ссылке: <https://www.atlassian.com/ru/git/tutorials/saving-changes/gitignore>

#### <p style="font-size: 15px;">Порядок действий</p>

<p style = text-indent:25px>Команды git add и git commit составляют основу рабочего процесса Git. Команды используются для записи версий проекта в историю репозитория.</p>

<p style = text-indent:25px>Работа над проектом ведется по стандартной схеме «редактирование — индексирование — коммит». Сначала вы редактируете файлы в рабочем каталоге. Когда вы будете готовы сохранить копию текущего состояния проекта, вы индексируете изменения командой git add. Затем вы вызываете команду git commit, которая добавляет проиндексированный снимок состояния в историю проекта. </p>

<br><a name="log"></a>

### <p style="font-size: 20px;">Просмотр версий</p>

```
git log
```

Выводит перечень версий. Если коммитов много,  перемещаться по ним можно стрелочками вверх и вниз. Пример вывода команды `git log`:

```
commit 0aaddda8b047e54c9cd856c27098625f85edc9d2
Author: Nikolay <nikolay_git@rambler.ru>
Date:   Sun Jun 12 11:17:17 2022 +0500

Добавлена команда git init
```

<br>
<a name="version-change"></a>

### <p style="font-size: 20px;">Перемещение между версиями файлов</p>

```
git checkout первые 4-6 символов commita
```

Позволяет перемещаться на разные состояния файлов.

Пример:

```        
git checkout 0aaddd
```

вернет состояние файла на 12 июня 11:17 часов  2022г 

0aaddd - первые 6 символов коммита полученного командой `git log`
>commit 0aaddda8b047e54c9cd856c27098625f85edc9d2"

Для быстрого перемещения к последнему состоянию файла используется команда:
```
git checkout master
```

<a name="branch-work"></a>

## <p style="font-size: 30px;">Работа с ветками</p>
---
###### <p align="right">[Cодержание](#content)</p>
<br>
<p style = text-indent:25px> Возможность ветвления доступна в большинстве современных систем контроля версий. Однако эта операция в ряде систем может быть довольно затратной как по времени, так и по объему дискового пространства. В Git ветки — это элемент повседневного процесса разработки. По сути ветки в Git представляют собой указатель на снимок изменений. Если нужно добавить новую возможность или исправить ошибку (незначительную или серьезную), вы создаете новую ветку, в которой будут размещаться эти изменения.</p>

<br><a name="branch-name"></a>

### <p style="font-size: 20px;">Создание ветки</p>
```
git branch имя_новой_ветки
```

<br><a name="branch-m"></a>

### <p style="font-size: 20px;">Переименование веток</p>

```
git branch имя_ветки -m "новое_имя_ветки"
```

<br><a name="branch-d"></a>

### <p style="font-size: 20px;">Удаление веток</p>
```
git branch -d имя_удаляемой_ветки  
или 
git branch -D имя_удаляемой_ветки
```

`git branch -d имя_удаляемой_ветки` Это «безопасная» операция, поскольку Git не позволит удалить ветку, если в ней есть неслитые изменения.

`git branch -D имя_удаляемой_ветки` Принудительное удаление указанной ветки, даже если в ней есть неслитые изменения. Эта команда используется, если вы хотите навсегда удалить все коммиты, связанные с определенным направлением разработки.

<br><a name="branch"></a>

### <p style="font-size: 20px;">Просмотр созданных веток</p>
```
git branch
```
Пример вывода данной команды:
```
* branch_work
  commands_git
  history_reference
  master
```
В данном случае "*" отмечена ветка в которой мы находимся.

<br><a name="checkout-branch"></a>

### <p style="font-size: 20px;">Переходы между ветками</p>
```
git checkout имя_ветки
```
<p style = text-indent:25px>Переходы между ветками осуществляются аналогично переходам между коммитами, с той лишь разницей, что вместо первых символов коммита прописывается имя ветки, в которую мы хотим попасть.</p>

<br> <a name="merge-branch"></a>

### <p style="font-size: 20px;">Слияние веток</p>
```
git merge имя_ветки
```
Данная команда производит перенос данных из одной ветки в другу.

**Важно:** *Команда перенесет данные из указанной ветки в ветку из которой эта команда запущена.*

#### <p style="font-size: 15px;">Конфликты при слиянии</p>

<p style = text-indent:25px>Конфликт возникает, когда в двух ветках была изменена одна и та же строка в файле или когда некий файл удален в одной ветке и отредактирован в другой. Как правило, конфликты возникают при работе в команде.</p>

<p style = text-indent:25px>В данном случае нужно будет определить как правильно отредактировать файл самостоятельно.</p>

<a name="online-rep"></a>

### <p style="font-size: 20px;"> Создание удаленных репозиториев </p>

<p style = text-indent:25px> Воспользуемся  крупнейшим веб-сервисом для хостинга IT-проектов и их совместной разработки - Github.com.</p>

<p style = text-indent:25px>Для начала работы необходимо создать аккаунт, или воспользоваться уже имеющимся.</p>

1. На сайте [Github.com](https://github.com/) на странице своего аккаунта нажимаем на "+" в правом-верхнем углу. Выбираем "New repository".
![Рисунок 1](/screenshots/Screen_1.jpg)
2. Заполняем обязательное поле "Repository name" и нажимаем "Create repository".
![Рисунок 1](/screenshots/Screen_2.jpg)
3. Создаем локальный репозиторий и делаем как минимум один коммит [подробнее об этом](#commit).
4. Воспользуемся набором команд для отправки документов из локального репозитория в удаленный:
```
git remote add origin url
git branch -M main
git push -u origin main
```
###### при первом использовании команды "push" потребуется пройти авторизацию на github.

### Разберем команды подробнее:
`git remout add origin url` - связывает локальный репозиторий и удаленный. На место url подставляется ссылка полученная на сайте Github.com.
![Скриншот нахождения ссылки]()

`git branch -M main` - указывает какую ветку локального репозитория считать основной.

`git push -u origin main` - направляет данные в удаленный репозиторий.

5. После проделанных выше действий достаточно будет команды `git push` для отправки данных в репозиторий.

<br>

<a name="load-file"></a>

### <p style="font-size: 20px;"> Скачивание данных с удаленного репозитория.</p>

1. На сайте [Github.com](https://github.com/), выбираем интресующий нас проект и делаем его Fork.
###### Fork - перетаскивание репозитория с чужого аккаунта на свой.
2. Выполняем команду
```
git clone url_репозитория
```
<p style = text-indent:25px>Данная команда создает локальный клон удаленного репозитория.</p>

<p style = text-indent:25px>В случае если нам нужно только получить последную версию удаленного репозитория необходимо выполнить команду</p>
```
git pull
```
Данная команда скачивает данные с выбранного репозитория.

<br>

**Важно**: 

Команда `git pull` является составной командой и выполяет 2 действия.
1. Выкачивает файлы с репозитория.
2. Объединяет (merge) скачиваемые файламы с файлами локального репозитория.

<br>
<a name="pull-request"></a>

### <p style="font-size: 20px;"> Внесение изменений в чужой проект.</p>

Если мы хотим предложить изменения в чужом проекте:
1. [Создаем локальную копию удаленного репозитория](#load-file)
2. [Создаем новую ветку в локальной копии](#branch)
3. В свой аккаунт заливаем внесенные изменения, команда `git pull`
4. На сайте Github.com в Вашем аккаунте появляется кнопка  отправить изменения автору проекта - `pull request`
![Кнопка отправить изменения]()

<p style = text-indent:25px> Автор проекта получит предложенные Вами изменения, после чего примет решение о необходимости добавления изменений в свой проект.</p>
